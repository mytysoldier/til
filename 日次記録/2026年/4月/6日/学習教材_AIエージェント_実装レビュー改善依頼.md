## 2. 事前知識チェック（目安時間：**5分**）

**Q1. 「実装依頼」と「レビュー依頼」で、AIに渡すべき情報の量はどちらが多くなることが多いか？**  
**A.** 一般に **レビュー依頼**の方が多い。動く文脈（目的・制約・入出力・再現手順・既知の懸念）を揃えないと、レビューは抽象論に落ちやすい。

**Q2. 「改善依頼」で最も事故りやすいのは何か？**  
**A.** **成功条件が曖昧**なまま「もっと良くして」と頼ること。結果としてスコープが膨らみ、仕様が勝手に変わる。

**Q3. 指示の粒度を上げすぎると何が失われがちか？**  
**A.** AIの**探索（設計の候補出し）**や、人間が気づきにくい**代替案の提示**が減る。全部手順化すると「実装代行」になりやすい。

## 3. 理論（目安時間：**15分**）

### ポイント1：3種類は「工程」ではなく**責務の切り分け**

- **実装依頼**：要件→コードへ（生成・初版づくり）
- **レビュー依頼**：既存成果物→リスク・設計・可読性・テスト観点へ（批評と優先順位）
- **改善依頼**：特定の欠点/指標→差分で直す（性能、境界条件、API、運用性など）

**よくある誤解/落とし穴**：レビュー依頼なのにコードが無い／再現できない。レビューは「読む対象」と「評価軸」が無いと空転する。

---

### ポイント2：AIに任せる責務は「広い探索」と「下書き」、人間が握るのは**境界と採否**

- AI：候補実装、レビューコメント草案、リファクタ案の列挙
- 人間：スコープ、互換性ポリシー、セキュリティ境界、リリース判断、採用する設計の決定

**よくある誤解/落とし穴**：「AIが通った＝安全」。特に入力検証・権限・秘密情報は人間側のチェックリストが要る。

---

### ポイント3：指示の粒度は**レビュー＞改善＞実装（初回）**になりやすい

- 初回実装は要件を固めつつ、探索余地を残す
- レビューは観点（例：境界条件、エラー設計、命名、テスト網羅）を明示
- 改善は「測れるゴール」（例：この入力で落ちない、複雑度を下げる、公開APIを変えない）を固定

**よくある誤解/落とし穴**：実装依頼から過密な擬似コードまで書くと、AIの設計余地が消え、あなたの思考停止も起きやすい。

---

### ポイント4：同じ題材で3種類試すと、**プロンプトの型**が身体に付く

同じコードを素材にすると、3種類の差分が明確になる（何を添えると品質が上がるかが学べる）。

**よくある誤解/落とし穴**：題材が大きすぎると「実装依頼」だけで時間が溶ける。**60分なら100行未満**が目安。

---

### ポイント5（設計の選択肢）：「スコアを整数で返す」vs「構造化結果を返す」

- **整数スコア**：CLI表示が簡単、ただし失敗理由が説明しづらい
- **構造化（理由付き）**：UX/テストが楽、ただしAPIが膨らむ

**なぜこの教材では整数から始めるか**：最初の対話で「動く核」を作り、改善依頼で構造化へ移行する流れが、3種類の依頼の違いを学びやすい。

**よくある誤解/落とし穴**：最初から完璧な型設計を狙って実装依頼が長文化し、ハンズオンが終わらない。

---

### ポイント6：エラー・型・「状態」／非決定性の落とし穴（実務で効く）

- **型**：アノテーションは `str` でも、実行時に `bytes` や `None` が来うる。境界は「どう失敗させるか」（黙って0点／`TypeError` 等）を仕様に書く。
- **エラー**：例外を飲み込むとテストが「静かに壊れる」。ログ・戻り値・例外のどれで失敗を表すかを先に決める。
- **状態**：この題材はステートレスだが、**AIの出力は実行のたびに変わりうる**（同一プロンプトでも差分）。レビュー・改善の再現には、コード全文とプロンプト版のメモを残す。
- **非同期**：今回のコードは同期のみ。非同期コードをAIに頼むときは「キャンセル・タイムアウト・例外の伝播」を明示しないと、動くが危ない実装になりやすい。

### ステップ5：テスト（標準ライブラリのみ）— **不変条件**で実装差に強くする

**方針**：スコアの「期待値」を固定しないと、AI実装ごとにテストが落ちる。**型・範囲・空文字**など、仕様として言い切れる不変条件で断言する。

**手順**

1. `tutorial/test_password_strength.py` を作成する（ファイル名は `test_*.py` で `unittest` の discovery に乗る）。
2. 次の **最低3アサーション** を満たす（1つのメソッドにまとめても、分けてもよい）。
   - `score_password("")` が **int** であること、かつ **0〜100** に入ること。
   - ある程度長くクラスが揃ったパスワード文字列（例：`"Abcdef1!x"`）に対し、スコアが **0〜100** の **int** であること。
   - `self.assertRaises(TypeError, ...)` で **`None`** または **`123`**（非str）を渡したときに `TypeError` になること（実装が「黙って0点」なら、その仕様に合わせて **この項目を「採用しない」ではなく実装に合わせて断言を変える** —— 仕様とテストを一致させる）。
3. 実行（必ず `tutorial` がカレント）：

```bash
cd tutorial
python -m unittest discover -v -s . -p 'test_*.py'
```

**最小コード例（テンプレート）**

```python
import unittest
from password_strength import score_password

class TestPasswordStrength(unittest.TestCase):
    def test_invariants(self):
        self.assertIsInstance(score_password(""), int)
        self.assertGreaterEqual(score_password(""), 0)
        self.assertLessEqual(score_password(""), 100)
        s = score_password("Abcdef1!extra")
        self.assertIsInstance(s, int)
        self.assertGreaterEqual(s, 0)
        self.assertLessEqual(s, 100)

    def test_rejects_non_str_if_that_is_spec(self):
        with self.assertRaises(TypeError):
            score_password(None)  # 実装がTypeErrorでない場合は、このテストを実装に合わせて書き換える

if __name__ == "__main__":
    unittest.main()
```

**確認方法**

- 上記 `unittest` コマンドで **FAILED 0**（少なくとも上記の意図したテストが通る）。
- `Ran 0 tests` になる場合：カレントが `tutorial` か、`test_*.py` の命名かを確認する。

## 6. 実務での使いどころ（目安時間：**5分**）

1. **実装依頼**：新規サービスの **OpenAPI の雛形**や **CRUD のボイラープレート**を一気に出し、人間は **認可の境界**と **禁止したい依存**（例：特定SDKは使わない）で枠を固定する。
2. **レビュー依頼**：自分のPRの diff に加え、**本番インシデントの要約（発生時刻・影響ユーザー数・再現手順）**を貼り、「同系の穴が他にないか」を評価軸に含める。指摘は **上位2〜3件だけ** をマージ前に採用する。
3. **改善依頼**：**p99 レイテンシが SLA を超えた箇所のプロファイル結果**や **`npm audit` / `pip-audit` の該当パッケージ行**を添え、「この関数だけ」「この依存の更新だけ」と変更範囲を列挙して回帰を防ぐ。
