## 2. 事前知識チェック（目安: 5分）

### Q1. Swift の `DispatchQueue.main.async` と `async/await` の主な違いは何か？

<details>
<summary>回答</summary>

- **DispatchQueue**: コールバックベース。ネストが深くなりやすく、エラーハンドリングが分散する。
- **async/await**: 同期的な見た目で非同期を書ける。`try/throw` と統合され、エラー伝播が一貫する。構造化並行性（Structured Concurrency）により、タスクのライフサイクルが明確になる。

**コード例（API 取得 → UI 更新）**

```swift
// DispatchQueue.main.async（コールバック地獄になりやすい）
URLSession.shared.dataTask(with: url) { data, _, error in
    if let error = error {
        DispatchQueue.main.async { self.showError(error) }
        return
    }
    guard let data = data else { return }
    parseJSON(data) { result in
        DispatchQueue.main.async {
            switch result {
            case .success(let model): self.updateUI(model)
            case .failure(let err): self.showError(err)
            }
        }
    }
}.resume()

// async/await（同期的な見た目で書ける）
let data = try await URLSession.shared.data(from: url)
let model = try JSONDecoder().decode(Model.self, from: data)
await MainActor.run { updateUI(model) }  // または @MainActor の関数を await
```
</details>

### Q2. `@MainActor` とは何か？なぜ必要か？

<details>
<summary>回答</summary>

UI を更新するコードはメインスレッドで実行する必要がある。`@MainActor` は、その型や関数がメインスレッドで実行されることを保証する属性。Swift Concurrency では、`DispatchQueue.main.async` の代わりに `@MainActor` でメインスレッド実行を表現する。
</details>

### Q3. 「構造化並行性」とは何か？非構造化（unstructured）との違いは？

<details>
<summary>回答</summary>

**構造化並行性**: 子タスクが親のスコープ内で完結し、親が終わる前に子も終わる。`async let` や `TaskGroup` で表現。親がキャンセルされると子も伝播する。

**非構造化**: `Task { }` で作った「火付けっぱなし」のタスク。親の完了を待たず、ライフサイクルが独立。必要だが、多用するとリークや予測困難な動作の原因になる。
</details>

## 3. 理論（目安: 12分）

### 重要ポイント1: async 関数は「中断可能な関数」

- `async` は「この関数は `await` の地点で一時停止し、再開する」という宣言。
- **よくある誤解**: 「async = 別スレッドで動く」ではない。スレッドはランタイムが割り当てる。メインスレッド上で `await` する async 関数もある。
- **落とし穴**: `async` 関数は通常の同期コンテキストから直接呼べない。`Task { }` や `await` の付いた async コンテキストが必要。

### 重要ポイント2: Task の3つの役割

| 用途 | 書き方 | 特徴 |
|------|--------|------|
| 同期→非同期の入口 | `Task { await foo() }` | 戻り値は `Task<T, Error>`。`.value` で await 可能 |
| 並列実行 | `async let x = a(); async let y = b()` | 子タスクを並列に開始し、両方 await |
| 明示的なキャンセル | `Task { ... }.cancel()` | キャンセルは協調的。タスク内で `Task.isCancelled` をチェック |

- **よくある誤解**: `Task` は必ずバックグラウンドで動くわけではない。`@MainActor` のコンテキストで作ればメインスレッドで動く。
- **落とし穴**: `Task { }` で作ったタスクは「親を待たない」。View が破棄されてもタスクは動き続ける。`task` 修飾子（SwiftUI）を使うと View のライフサイクルに紐づく。

### 重要ポイント3: MainActor と UI 更新

- UI 更新はメインスレッド必須。`@MainActor` を付けた型・関数はメインで実行される。
- **よくある誤解**: `await MainActor.run { }` を毎回書く必要はない。`@MainActor` を付ければ `await` なしでメインに切り替わる。
- **落とし穴**: バックグラウンド Task から UI を更新するとクラッシュ。`@MainActor` の関数を `await` するか、`MainActor.run` で囲む。

### 重要ポイント4: エラー伝播と try/await

- `async throws` 関数は `try await` で呼ぶ。エラーは同期的な `throw` と同じように伝播する。
- **落とし穴**: `Task { }` 内の `throws` は外に伝播しない。`Task` の結果を `await` するか、`@Sendable` クロージャ内で適切にハンドリングする必要がある。
- **落とし穴**: キャンセル時は `CancellationError` が throw される。`catch is CancellationError` で意図的なキャンセルと区別する。

### 重要ポイント5: Sendable とデータ競合

- 並行処理間で共有するデータは `Sendable` に準拠すべき。コンパイラがデータ競合を検出する助けになる。
- **落とし穴**: クラスはデフォルトで非 Sendable。`@unchecked Sendable` は「自分で競合を防いでいる」とコンパイラに伝えるもので、安易に使うと危険。
- **落とし穴**: 非同期処理中にクラスの `var` を更新するとデータ競合のリスク。`actor` や `@MainActor` で排他制御する。

### 重要ポイント6: 設計の選択肢 — Task vs async let

- **Task**: 明示的に開始・キャンセルしたい、結果を後で使いたい場合。
- **async let**: 複数の非同期処理を並列に開始し、両方の結果をまとめて使う場合。構造化されており、親のキャンセルが子に伝播する。
- **今回のハンズオンでは両方扱う**: ステップ3で `Task`、ステップ4で `async let` を実装する。

```swift
// Sources/AsyncTutorial/Fetcher.swift
import Foundation

enum FetcherError: Error {
    case invalidURL
    case networkError(Error)
}

enum Fetcher {
    static func fetchData(from urlString: String) async throws -> Data {
        guard let url = URL(string: urlString) else {
            throw FetcherError.invalidURL
        }
        do {
            let (data, _) = try await URLSession.shared.data(from: url)
            return data
        } catch {
            throw FetcherError.networkError(error)
        }
    }
}
```

### ステップ4: 並列取得（async let）の追加（7分）

**async let とは**

複数の非同期処理を**同時に開始**し、後でまとめて `await` する構文。`Task` を2つ作るより簡潔で、構造化並行性により親のキャンセルが子に伝播する。

```swift
// async let: 両方の取得を並列に開始
async let a = Fetcher.fetchData(from: url1)
async let b = Fetcher.fetchData(from: url2)
// ここで初めて待つ（a と b は並列に実行中）
let (dataA, dataB) = try await (a, b)
```

**手順**

`tutorial/Sources/AsyncTutorial/main.swift` を次のように書き換え、単一取得（Task）と並列取得（async let）の両方を含める。

```swift
// Sources/AsyncTutorial/main.swift
import Foundation

await runSingleFetch()
await runParallelFetch()

func runSingleFetch() async {
    let urlString = "https://httpbin.org/get"
    let task = Task {
        try await Fetcher.fetchData(from: urlString)
    }
    do {
        let data = try await task.value
        print("単一取得: \(data.count) バイト")
    } catch {
        print("単一取得エラー: \(error)")
    }
}

func runParallelFetch() async {
    async let a = Fetcher.fetchData(from: "https://httpbin.org/delay/1")
    async let b = Fetcher.fetchData(from: "https://httpbin.org/delay/1")
    do {
        let (dataA, dataB) = try await (a, b)
        print("並列取得: A=\(dataA.count), B=\(dataB.count) バイト")
    } catch {
        print("並列取得エラー: \(error)")
    }
}
```

**確認方法**

- `swift run` で両方のメッセージが表示される
- 並列取得は約2秒で終わる（直列なら約4秒かかる想定）
- ※ `httpbin.org/delay/1` が遅い場合は、両方 `"https://httpbin.org/get"` に変更しても並列の動きは確認できる

**Task と async let の違い（このステップで学ぶ）**

| | Task | async let |
|--|------|-----------|
| 用途 | 単一タスク、後で結果を await | 複数タスクを並列開始し、まとめて await |
| 構造化 | 非構造化（親を待たない） | 構造化（親のスコープ内で完結） |
| キャンセル | 明示的に `.cancel()` | 親がキャンセルされると子も伝播 |

### Hard: TaskGroup を使って複数 URL を並列取得し、合計バイト数を返す関数を実装する

<details>
<summary>回答例</summary>

```swift
func fetchTotalBytes(urls: [String]) async throws -> Int {
    try await withThrowingTaskGroup(of: Data.self) { group in
        for urlString in urls {
            group.addTask {
                try await Fetcher.fetchData(from: urlString)
            }
        }
        var total = 0
        for try await data in group {
            total += data.count
        }
        return total
    }
}
```
</details>

## 6. 実務での使いどころ（具体例3つ・目安: 5分）

### 例1: API クライアント層

// 例: UserRepository
func fetchUser(id: String) async throws -> User {

### 例2: SwiftUI の `task` 修飾子

`List` や `DetailView` で、表示時にデータを取得する場合。

```swift
struct UserListView: View {
    @StateObject var viewModel = UserListViewModel()
    var body: some View {
        List(viewModel.users) { ... }
            .task { await viewModel.load() }  // View 破棄時に自動キャンセル
    }
}
```

### 例3: バックグラウンド処理 + メインスレッドでの UI 更新

重い計算後に UI を更新する場合。

```swift
// 例: 画像リサイズ
Task {
    let resized = await resizeImage(heavyImage)  // バックグラウンド
    await MainActor.run { self.thumbnail = resized }  // メインで UI 更新
}
```

## 7. まとめ（今日の学び3行・目安: 2分）

- **async/await** は「中断・再開」の宣言であり、`await` のたびにスレッドをブロックせずに制御を戻す。
- **Task** は同期→非同期の入口であり、`async let` は構造化された並列実行。用途に応じて使い分ける。
- **@MainActor** で UI 更新をメインスレッドに限定し、`task` 修飾子で View のライフサイクルに合わせたキャンセルを活用する。
